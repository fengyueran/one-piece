# 领域陷阱研究

**领域：** 公开发布的 React 组件库 / 个人长期复用资产 monorepo  
**聚焦：** `compass-ui`  
**研究日期：** 2026-04-13

## 当前仓库最该提前规避的坑

结合当前仓库状态，最需要优先处理的不是“还缺多少组件”，而是“现有对外承诺是否真实成立”。眼下最危险的三个点是：

1. **文档与真实导出面不一致。** `@xinghunm/compass-ui` 当前只导出根入口 `.`，但文档仍直接引用 `@xinghunm/compass-ui/dist/locale/*` 与 `@xinghunm/compass-ui/src/*`。这会让文档示例在 monorepo 内可跑、在真实 npm 消费场景失效。仓库证据见 `packages/compass-ui/package.json:9-18` 与 `packages/compass-ui/docs/components/config-provider.md:31-33`、`date-picker.md:199,265`、`tree-select.md:303`、`auto-complete.md:428`。
2. **发布策略意图不单一。** 根级 Changesets 配置写的是 `access: "restricted"`，包级 `publishConfig.access` 却写 `public`，而根 `release` 脚本直接串行执行 lint/test/build 后 publish。对一个要公开发布的 scoped 包来说，这种配置冲突会在首次发布、换账号、切 CI 时制造不必要事故。仓库证据见 `.changeset/config.json:3-10`、`packages/compass-ui/package.json:74-76`、`package.json:12-16`。
3. **复杂交互组件先自研、后补规范。** `Select` 已有键盘导航缺口，`Floating UI` 相关组件还存在依赖规则豁免的脆弱实现。对于 `Select`、`TreeSelect`、`DatePicker`、`Dropdown` 这类组件，真正难的不是渲染，而是焦点管理、键盘交互、ARIA 语义和回归测试。如果在“补齐组件覆盖”阶段继续把复杂行为都自己扛，后面会进入越写越不敢动的状态。

建议把 roadmap 的前几阶段收敛为以下顺序：

- **Phase 1：公开 API 与发布基线收口**
- **Phase 2：文档、示例、验证链路对齐**
- **Phase 3：高频基础组件补齐**
- **Phase 4：复杂交互组件稳态化**

下面的每个坑都按这个阶段建议落位。

## 关键陷阱

### 1. 组件库范围失控，先开目录再补质量

**会出什么问题：**  
组件数量增长很快，但每个组件都只做到“能渲染 demo”，没有达到可公开发布的质量线。最后得到的是一个看起来覆盖面不少、实际没有几个能放心复用的库。

**为什么常发生：**  
组件库最容易被“清单驱动”带偏。看到成熟库里有多少组件，就倾向于照着补；但成熟设计系统普遍不是按“想到了就收编”，而是按复用证据、维护成本和成熟度门槛来决定是否纳入。Primer、GitLab Pajamas、GOV.UK 都明确要求：新组件必须证明跨场景复用价值，不能只是单点需求。

**当前仓库触发条件：**  
`PROJECT.md` 已明确“不追求 Ant Design 级覆盖”，但同时又把“补齐常用组件能力”作为主线。如果没有一套明确的准入标准，后续最容易从“补常用组件”滑向“持续补齐成熟库目录”。

**预警信号：**

- 新组件需求的描述是“这个库里也有，所以我们也加一个”，而不是“我自己已有 2 个以上真实复用场景”
- 组件进入包之前，没有“是否已有变体可覆盖”的审查
- 文档页数量增长快于稳定 API 数量
- 出现多个只服务单一业务语境的组件，但仍被放进 `compass-ui`

**避免策略：**

- 建立最小准入标准：只有满足“高频复用、跨项目适用、可维护、命名稳定”的需求才进入 `compass-ui`
- 先补“Button/Input/Form/Overlay/Data Entry”这类高频基础件，不做低频炫技件
- 对新增组件引入“先局部验证，再入库”的成熟度规则：先在 demo 或本地项目使用，再决定是否进入公开包
- 明确“变体优先于新组件”：先看能否通过现有组件扩展 props / slots / composition 解决

**建议在哪个 Phase 处理：**  
**Phase 1：公开 API 与发布基线收口**  
原因：范围规则必须先立，否则后面所有 phase 都会被新组件插队打断。

### 2. 文档与真实导出不一致，导致“仓库内可用、npm 用户不可用”

**会出什么问题：**  
对外文档成为错误示例库。用户按文档复制代码后构建失败，或者依赖内部路径，后续一发版就 break。

**为什么常发生：**  
组件库作者通常同时维护源码、docs 站和 demo，仓库内部 alias 很方便，于是会在文档里不自觉地引用 `src/`、`dist/` 或临时导出。Node 官方文档明确说明：`exports` 字段定义的是包的公开接口；未列入 `exports` 的子路径不应被消费者依赖。

**当前仓库触发条件：**

- `@xinghunm/compass-ui` 当前 `exports` 只有 `"."`
- 文档示例仍引用 `@xinghunm/compass-ui/dist/locale/*`
- 文档示例仍引用 `@xinghunm/compass-ui/src/*/types`
- `.dumirc.ts` 里还对 `dist/locale` 做了本地别名补偿

**预警信号：**

- 文档里出现 `src/`、`dist/`、相对到包源码的导入
- README / docs 示例不能直接在一个空消费项目里运行
- 新增文档时先改 alias，后改公开导出
- 类型示例依赖内部 `types.ts`，而不是包根公开类型

**避免策略：**

- 规定文档示例只允许使用包公开导出路径；任何内部路径一律视为缺陷
- 建立“消费者冒烟测试”：用临时项目安装打包产物，只跑文档示例中的导入
- 如果确实需要公开 `locale` 或类型子路径，就正式加入 `exports`，而不是让 docs 走后门
- 文档更新和导出面更新必须同一个 PR 完成，避免分离演进

**建议在哪个 Phase 处理：**  
**Phase 1：公开 API 与发布基线收口**  
原因：这是当前最直接影响公开 npm 可用性的坑，应该先于新增组件。

### 3. 测试投入失衡，只堆单测覆盖率，不验证真实消费链路

**会出什么问题：**  
包内测试分数很好看，但发布包、文档站、示例应用、无障碍行为和真实安装路径照样出问题。最终测试投入变成“保护实现细节”，而不是“保护公开契约”。

**为什么常发生：**  
组件库最容易对单文件覆盖率上瘾，因为它最好量化。Material UI 明确建议用接近用户的方式测试，而不是绑定内部实现；Storybook 和 VA.gov 的实践也都强调，自动化 a11y 检查只是第一层，仍需要针对真实 DOM、键盘交互和集成场景做验证。

**当前仓库触发条件：**

- `compass-ui` 和 `compass-hooks` 有很高覆盖率阈值，但应用层和文档层几乎没有测试
- 根 `pnpm test` 不覆盖 `apps/docs` 的页面行为，也不覆盖 `apps/ai-chat-demo` 主接线
- 当前已有无障碍缺口：`Select` 缺少键盘导航
- 发布链路没有“文档示例可消费”和“导出面完整性”自动校验

**预警信号：**

- 讨论测试时只看 `coverageThreshold`，不看“消费者会怎么用”
- 一个组件有很多渲染分支测试，但没有键盘交互测试
- 文档页改坏了只能靠手点
- 发布前没有 `npm pack` / 临时安装 / docs 示例编译验证

**避免策略：**

- 把测试拆成四层：组件行为单测、无障碍/键盘交互测试、文档/示例冒烟测试、发布包消费测试
- 对高风险组件只要求“关键路径高信心”，不要为了覆盖率去测试内部 className 和实现细节
- 复杂交互组件至少加入 keyboard + focus + axe 三类检查
- 用少量高价值的 consumer smoke tests 替代大量低价值快照

**建议在哪个 Phase 处理：**  
**Phase 2：文档、示例、验证链路对齐**  
原因：测试策略应围绕公开消费链路重构，而不是等组件更多后再补。

### 4. 发布策略混乱，导致版本、访问级别、发布节奏不可预测

**会出什么问题：**  
首次发布失败、公开包被当私有包处理、版本 bump 与内部依赖关系不匹配、发布事故只能靠人工经验排查。

**为什么常发生：**  
monorepo 里的发布配置常常是“工具默认值 + 包级补丁 + 人工记忆”的叠加。Changesets 官方文档对 `access`、`fixed`、`linked`、`ignore` 都给了明确语义；npm 官方也说明 scoped package 首次公开发布需要明确 `--access public` 或等效配置。配置源头一旦不单一，后续每次换环境都可能踩雷。

**当前仓库触发条件：**

- `.changeset/config.json` 写 `access: "restricted"`
- `packages/compass-ui/package.json` 写 `publishConfig.access: "public"`
- 根 `release` 脚本没有 pack/安装/导出校验，只是在 packages 上跑 lint/test/build 后直接 publish
- `fixed` / `linked` 都为空，说明多包之间的版本策略尚未被明确建模

**预警信号：**

- 团队自己都说不清“以根配置为准还是包配置为准”
- 发布规则只能靠“我记得上次是这样发的”
- 修改内部依赖关系后，没有人确认 version bump 是否符合预期
- 发布成功与否高度依赖本地 npm 账号状态和手工参数

**避免策略：**

- 明确唯一发布真相源：决定是以 Changesets 全局 `access` 为准，还是包级 `publishConfig` 为准，然后消灭另一套歧义
- 补一个 `release:check`：至少做 `changeset status`、`npm pack`、临时安装、公开导出检查
- 明确哪些包应该联动版本、哪些包允许独立发版
- 首次公开发布前，把 npm 可见性、tag 策略、2FA/token 策略写进仓库文档

**建议在哪个 Phase 处理：**  
**Phase 1：公开 API 与发布基线收口**  
原因：发布策略不稳定时，越往后补组件，回滚成本越高。

### 5. 过早做差异化，跳过 table stakes

**会出什么问题：**  
组件库在还没把基础交互、无障碍、文档和 API 稳定性做好之前，就去追求“我们和别人不一样”，结果既没有差异化价值，也没有基础可信度。

**为什么常发生：**  
个人项目很容易被“为什么用户要用我”这个问题逼着提前找定位。但成熟设计系统的共识是：先证明组件在多个真实场景可用，再谈是否值得推广、沉淀或上游。Primer 和 GOV.UK 的准入标准都把“可复用、可维护、已验证”放在“值得推广”前面。

**当前仓库触发条件：**  
`PROJECT.md` 已明确“先补齐常用能力，再考虑差异化”，这本身是对的；真正的风险在于实现时又被新颖需求或品牌化冲动打断。

**预警信号：**

- 开始优先做少见但“很有辨识度”的组件，而不是表单、选择器、反馈类基础件
- 为了设计风格差异，牺牲与平台习惯一致的交互
- 文档强调理念多于可用示例
- 基础组件还不稳定，就开始讨论 design token 平台化、组件元数据平台、生成式规范系统

**避免策略：**

- 把差异化定义收窄为“默认设计语言和可组合方式”，而不是“重新发明交互模式”
- 先把高频基础组件做到可靠，再在视觉层和组合层做独特表达
- 每个新方向都问一句：这会不会推迟一个用户更早就会遇到的基础问题

**建议在哪个 Phase 处理：**  
**Phase 3：高频基础组件补齐**  
原因：这个阶段最容易被“做点有意思的”带偏，需要明确克制。

### 6. 自研与复用开源库判断失真，把“能写”误判成“值得长期维护”

**会出什么问题：**  
对复杂组件自己实现核心交互逻辑，前期看起来自主可控，后期却持续为焦点管理、键盘导航、屏幕阅读器、国际化和边界行为付维护税。

**为什么常发生：**  
写出一个能用的 `Select` 不难，写出一个长期稳定的 `Select` 很难。WAI-ARIA APG 对 combobox/listbox 的键盘行为要求非常细；Radix 和 React Aria 之所以存在，就是因为这些行为细节并不适合每个团队都从头维护。它们的官方定位都很明确：提供无样式、可组合、带可访问性行为的基础 primitives，让设计系统把精力花在 API 和样式表达上。

**当前仓库触发条件：**

- `Select` 已经缺少键盘导航
- `TreeSelect`、`DatePicker`、`Dropdown`、`Tooltip` 都属于焦点和 overlay 复杂度高的区域
- 代码关注点文档显示多个 Floating UI 相关组件依赖规则豁免和脆弱实现

**预警信号：**

- 每个复杂组件都在重复实现焦点回收、`aria-activedescendant`、键盘循环导航
- 一改 overlay 行为就要同时改大量回归测试
- 为了继续推进功能，不断添加 `eslint-disable react-hooks/rules-of-hooks` 或 `@ts-expect-error`
- 组件 API 设计被底层技术债牵着走

**避免策略：**

- 对“行为复杂、样式自由度高”的组件优先采用成熟 primitives / hooks，而不是全自研
- 自研只保留在真正能形成稳定公共 API 的层：命名、组合方式、视觉系统、业务友好默认值
- 把“依赖第三方 primitive”视为缩小维护面，而不是失去控制权
- 为每个复杂组件先做一张“哪些能力自己维护、哪些能力交给基础库”的边界表

**建议在哪个 Phase 处理：**  
**Phase 4：复杂交互组件稳态化**  
原因：这是在补齐基础组件之后最值得集中投入的技术分水岭。

## 中度陷阱

### 7. 没有组件成熟度分层，所有组件一发布就默认“稳定可用”

**会出什么问题：**  
用户不知道哪些组件只是可试用、哪些组件已经稳定；维护者也无法用公开状态管理预期，只能在 README 里一视同仁地承诺。

**当前仓库触发条件：**  
目前 `compass-ui` 更像单一成熟度对外暴露，没有“实验 / 可谨慎使用 / 稳定”的明显分层。

**预警信号：**

- 新组件一进主入口就默认和旧组件同级承诺
- README / 文档没有标记组件成熟度
- 维护者希望快速公开，但又担心 API 后续会频繁变

**避免策略：**

- 建立轻量成熟度标签：`experimental`、`beta`、`stable`
- 复杂组件在达到 keyboard + docs + smoke test + 真实使用前，不进入 `stable`
- 文档上明确标注 breaking risk 和适用范围

**建议在哪个 Phase 处理：**  
**Phase 2：文档、示例、验证链路对齐**

### 8. 用“新版本并存”掩盖迁移问题，长期制造双轨组件

**会出什么问题：**  
为了快速修一版复杂组件，新增 `NewSelect` / `NextModal` 之类替代件，短期绕开重构，长期让文档、导出和使用建议全部分裂。GitLab Pajamas 明确提到，这类“New 组件并存”会制造选型混乱和持续技术债。

**当前仓库触发条件：**  
虽然仓库里还没明显出现 `New*` 组件命名，但复杂组件已处于脆弱带，后续非常容易走这条看似省事的路。

**预警信号：**

- 讨论方案时频繁出现“要不新做一个，旧的先别动”
- 新旧组件长期共存但没有迁移计划
- 文档里开始出现“推荐用 A，但 B 还没废弃”

**避免策略：**

- 非必要不走双轨；优先在原组件内收敛行为和 API
- 如果必须双轨，发布时就写清迁移计划、替代关系和下线窗口
- 不允许无截止日期的临时 fork 常驻公共入口

**建议在哪个 Phase 处理：**  
**Phase 4：复杂交互组件稳态化**

### 9. 文档站被当成展示站，而不是发布契约的一部分

**会出什么问题：**  
文档内容更新滞后、示例只追求好看不追求真实、版本切换后旧文档仍暗示错误用法，最终让 docs 成为营销资产而不是工程资产。

**当前仓库触发条件：**  
当前 docs 使用 dumi，但缺少针对 docs 页面本身的自动验证，也存在通过 alias 绕过公开导出的情况。

**预警信号：**

- demo 为了展示效果写了大量只适用于站内环境的代码
- 文档没有和发版一起校验
- 用户问题更多来自“照文档做不通”，而不是“功能不支持”

**避免策略：**

- 把文档示例视为 public contract，不是参考草图
- 每次发版前至少验证 docs 示例里的关键导入路径
- 对每个组件只保留少量高价值示例，避免维护过多低信噪比 demo

**建议在哪个 Phase 处理：**  
**Phase 2：文档、示例、验证链路对齐**

## 轻度陷阱

### 10. 过早建设重型设计系统平台能力

**会出什么问题：**  
在组件本身还不稳定时，把时间投入到 token 平台化、设计工具同步、复杂元数据系统，导致“平台看起来很完整，组件却不够可靠”。

**当前仓库触发条件：**  
`PROJECT.md` 已明确当前不优先做复杂设计系统平台能力，这个边界非常正确，但后续在公开化过程中容易再次被包装需求诱惑。

**预警信号：**

- 讨论主题从“这个组件是否可靠”转向“要不要先做设计资产同步”
- 平台类脚手架和元数据先于组件 API 稳定

**避免策略：**

- 维持当前边界：组件可靠性优先于平台化
- 把平台建设设为明确的后续 milestone，而不是穿插在当前主线里

**建议在哪个 Phase 处理：**  
**暂不进入当前 milestone**  
只有当基础组件、文档、发布和消费验证已经稳定后再考虑。

## Phase 级别预警

| Phase 主题                        | 最容易踩的坑                                     | 缓解建议                                                        |
| --------------------------------- | ------------------------------------------------ | --------------------------------------------------------------- |
| Phase 1：公开 API 与发布基线收口  | 继续允许 docs 走内部路径；发布配置继续双重真相源 | 先统一 `exports`、子路径策略、Changesets 与 npm access 策略     |
| Phase 2：文档、示例、验证链路对齐 | 只修文档文案，不修文档执行环境和消费验证         | 建立文档导入校验、消费者冒烟测试、基础 a11y 检查                |
| Phase 3：高频基础组件补齐         | 为了“看起来更完整”一次性并行开太多组件           | 强制组件准入标准，只做高频基础件，其他进入 backlog              |
| Phase 4：复杂交互组件稳态化       | 继续全自研复杂行为，或用 `New*` 双轨组件逃避迁移 | 评估 Radix / React Aria 等 primitives，必要时重画边界而不是硬扛 |

## 研究结论

对 `compass-ui` 这种处于“准备公开、但仍在补齐基础能力”的组件库来说，真正的失败路径通常不是“组件做少了”，而是**在基础契约没有收口之前就开始扩张**。行业里成熟的设计系统几乎都在强调三件事：

1. **新增组件门槛要高。** 先证明复用价值，再决定是否收编。
2. **文档、代码、设计、测试要同步。** 否则用户最先碰到的是不一致，而不是能力不足。
3. **复杂行为优先复用成熟基础层。** 可访问性和交互细节不适合在个人长期维护的组件库里反复重造。

这也意味着，当前仓库最优节奏不是“立刻补更多组件”，而是先把公开边界、文档可信度、验证链路和复杂组件策略收紧，再进入下一轮组件补齐。
